When an offline metric lies, look at the join keys first

A model can improve in evaluation and still disappoint in launch. Before you blame ranking theory, inspect the plumbing.

Most offline-to-online gaps I have debugged started with an innocent assumption: the rows used for training, scoring, or attribution represented the same product reality. They often did not.

Start With Identity

Check whether the entity key means the same thing across logs, feature tables, labels, and serving traces. Versioned catalogs, marketplace boundaries, merged accounts, and delayed event streams can quietly turn a clean metric into a measurement of join luck.

Then Check Time

Freshness bugs are generous to offline evaluation. If a feature was available at training time but not at request time, your validation set may be testing a system you never actually serve.

Keep a Small Audit Query

For every launch, I like a boring query that samples examples from each stage and prints keys, timestamps, feature versions, label windows, and serving decisions in one place. It saves meetings.
